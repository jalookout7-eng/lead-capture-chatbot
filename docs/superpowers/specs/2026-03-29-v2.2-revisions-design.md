## Overview

Five changes to the live system plus a handover documentation update for future email automation.

## 1. Chatbot System Prompt Rewrite

**Problem:** The chatbot is too verbose, uses filler language, offers unsolicited suggestions, and subtly pitches services. Conversations run too long.

**Goal:** A direct, no-fluff conversational agent that strictly collects information about the prospect's business, current structure, and bottlenecks. It acknowledges what the prospect says (to show it's listening) but never gives advice, suggestions, or pitches. The team handles selling.

### New System Prompt Rules

- Ask ONE question per message normally, but can ask up to TWO questions if they're naturally related (e.g. "What does your business do and who do you serve?"). Never more than two.
- Acknowledge briefly ("Got it", "Makes sense", "Understood") then ask the next question. No elaboration, no suggestions, no "that's interesting because..." filler.
- Never offer advice, recommendations, or frame anything as a suggestion.
- Never pitch, sell, or hint at what 3D Visual Pro could do for them.
- Never ask about budget or timeline.
- If the visitor asks questions about 3D Visual Pro's services, pricing, or anything off-topic, acknowledge briefly but redirect: make it clear the chatbot's purpose is to learn about the visitor's business and reason for visiting, not to discuss those topics. Example: "I appreciate the question, but I'm here to learn about your business first. The team can go into all of that with you directly."
- Route into 4 discovery paths (AI Automation, Modern Websites, Digital Marketing, General/Not Sure) through natural conversation — no menu.
- Keep the total conversation to ~5-6 exchanges before CAPTURE_READY.
- Minimum 3 user-assistant exchanges before CAPTURE_READY can fire (reduced from 4). This remains a prompt-only guard (no server-side enforcement), consistent with v2 approach.

### Discovery Paths (streamlined)

Note: Each path has 3 core questions (down from 4 in v2). Combined with the opening question ("what brings you here"), this gives 4 total exchanges minimum — enough for AI qualification while keeping the conversation tight. This is intentional.

Each path asks direct questions to extract:
1. What the business does and who they serve
2. How they currently operate (tools, channels, processes)
3. Where things break down or slow down (bottlenecks)

**AI Automation path:**
- What does your business do?
- What tools/systems do you currently use day-to-day?
- Where do things slow down or fall through the cracks?

**Modern Websites path:**
- What does your business do?
- Do you have a website? What's not working about it?
- What's the main goal — leads, sales, or information?

**Digital Marketing path:**
- What does your business do?
- What marketing channels are you using right now?
- What's working and what's not?

**General / Not Sure path:**
- What does your business do?
- What's taking up most of your team's time?
- Where do things tend to slow down?

### Signal Behavior (unchanged)
- PRODUCT:<type> emitted once when path is identified
- CAPTURE_READY emitted at end of message after minimum exchanges and discovery is complete
- Both stripped server-side before reaching the user

### File Changed
- `src/routes/chat.js` — SYSTEM_PROMPT constant rewritten

## 2. Admin — Manual Lead Entry & Excel Upload

**Problem:** The admin dashboard currently only shows leads captured via the chatbot. The team needs to add leads from other sources (referrals, events, cold outreach).

### 2a. Manual Single Lead Form

A form accessible from the Leads tab with fields:
- **Name** (required)
- **Email** (required)
- **Phone** (optional)
- **Product** — dropdown: ai_service, website, marketing, consultancy, other (required)
- **Score** — dropdown: hot, warm, cold (required)
- **Notes** (optional textarea)

No sessionId, no AI qualification. Summary and bottlenecks fields will be empty. Followup will be empty.

**Email validation:** Basic format check (must contain @). No duplicate detection — the team may intentionally add the same contact multiple times for different products.

**API:** New endpoint `POST /api/leads/manual` (auth required). Accepts the fields above, generates UUID, sets status to "new", sets created_at to now.

### 2b. Excel Bulk Upload

An upload button in the Leads tab that accepts `.xlsx` or `.csv` files.

**Expected column format (order matters):**
| Column | Field | Required |
|--------|-------|----------|
| A | name | yes |
| B | email | yes |
| C | phone | no |
| D | product | yes |
| E | score | yes |
| F | notes | no |

**Processing:** Client-side parsing using the SheetJS (xlsx) library loaded from CDN. Each row validated, then sent to `POST /api/leads/import` (auth required) as a JSON array. Server inserts all valid rows, returns count of successes and any row-level errors.

**API:** New endpoint `POST /api/leads/import` (auth required). Accepts `{ leads: [...] }` array with a maximum of 500 rows per upload. Validates each entry (name, email with @ check, product, score required; score must be hot/warm/cold; product must be valid). Inserts valid rows one at a time — NOT transactional. If row 3 fails validation, rows 1-2 are already inserted and row 4+ continues. Returns `{ imported: N, errors: [{ row: N, reason: "..." }, ...] }`. This partial-success approach is intentional — the admin can fix and re-upload failed rows.

### UI Location
- "Add Lead" button and "Upload Excel" button at the top of the Leads tab, next to the search input.
- "Add Lead" opens a modal/inline form.
- "Upload Excel" opens a file picker, processes client-side, shows a confirmation count before submitting.

### Files Changed
- `src/routes/leads.js` — two new endpoints (POST /manual, POST /import)
- `public/admin/index.html` — UI for add lead form, upload button, file processing

## 3. Lead Detail — Explicit Save for Notes & Status Dropdown

**Problem:** Notes auto-save on blur (not obvious). Status has no UI control in the detail view.

### Changes
- Replace blur-based auto-save with an explicit **Save** button below the notes textarea.
- Add a **status dropdown** (new / contacted / converted / closed) in the lead detail view.
- Save button fires both PATCH calls in parallel (`Promise.all`). If either fails, show "Save failed" error. If both succeed, show "Saved" confirmation for 2 seconds.

### Files Changed
- `public/admin/index.html` — lead detail view updated

## 4. Quotes Feed — 5 Quotes with Fade Animation

**Problem:** Only 1 static quote shows per day.

### Design
- Display area shows one quote at a time.
- On page load, pick 5 random quotes from the existing 30-quote array (not sequential, not day-based).
- Each quote fades in (opacity 0 to 1, ~1s), holds for 4 seconds, fades out (opacity 1 to 0, ~1s), then the next quote fades in.
- Loops continuously through the 5 quotes.
- CSS transitions for the fade. JS setInterval or setTimeout chain to drive the cycle.
- Total cycle for all 5: ~30 seconds, then repeats.

### Files Changed
- `public/admin/index.html` — quote HTML container, CSS transitions, JS animation logic

## 5. Market Pulse — Expanded Rolling Ticker

**Problem:** Currently shows only BTC and ETH. NVIDIA was supposed to show but fails silently because Yahoo Finance blocks browser-origin requests (CORS). The current implementation fetches Yahoo directly from the browser with a try/catch that swallows the error.

### New Assets

**Stocks:**
- NVIDIA (NVDA)
- Tesla (TSLA)
- S&P 500 (^GSPC)

**Crypto:**
- Bitcoin (BTC)
- Ethereum (ETH)
- Solana (SOL)
- XRP (XRP)

**Commodities:**
- Gold (GC=F)
- Oil / WTI Crude (CL=F)

### Data Sources & Proxy

**Problem:** Yahoo Finance API does not set CORS headers, so browser-direct calls will be blocked. CoinGecko works from the browser but is rate-limited.

**Solution:** Add a server-side proxy endpoint `GET /api/market` (no auth required — this is non-sensitive public data). The Express server fetches all prices server-side and returns a unified JSON response. The admin dashboard calls this single endpoint.

**Server-side fetches:**
- **Crypto:** CoinGecko API — `https://api.coingecko.com/api/v3/simple/price?ids=bitcoin,ethereum,solana,ripple&vs_currencies=usd&include_24hr_change=true`
- **Stocks + Commodities + Indices:** Yahoo Finance chart API (unofficial, no SLA) — `https://query1.finance.yahoo.com/v8/finance/chart/{SYMBOL}?range=2d&interval=1d` for each of: NVDA, TSLA, ^GSPC, GC=F, CL=F

**Response format:**
```json
{
  "crypto": { "BTC": { "price": 70000, "change24h": 2.5 }, ... },
  "stocks": { "NVDA": { "price": 950, "change24h": -1.2 }, ... },
  "commodities": { "Gold": { "price": 2300, "change24h": 0.8 }, ... },
  "fetchedAt": "2026-03-29T12:00:00Z"
}
```

**Server-side caching:** Cache the response in memory for 1 hour. If the cache is fresh, return it immediately without re-fetching. This prevents rate-limit issues even with multiple admin sessions open.

**Fallback:** If a Yahoo Finance call fails (403, 429, timeout), that asset shows "N/A" in the ticker. If CoinGecko fails, all crypto shows "N/A". The ticker still renders with available data.

**New file:** `src/routes/market.js` — market proxy endpoint, registered in `src/server.js`.

### Loading & Error States
- On initial page load, ticker shows "Loading prices..." placeholder.
- If all API calls fail, ticker shows "Market data unavailable" static message.
- Individual assets that fail show "N/A" for price and no percentage.

### Rolling Ticker Design
- Vertical scrolling ticker that rolls bottom-to-top continuously.
- Each item shows: symbol/name, current price, 24hr % change (green if positive, red if negative).
- CSS animation: `@keyframes tickerScroll` translating Y from 100% to -100% (or similar).
- All items stacked vertically in a container that scrolls upward in a loop.
- Container is the same card area currently used by Market Pulse.

### Hourly Refresh
- On page load, fetch all prices.
- `setInterval` at 3600000ms (1 hour) to re-fetch all prices.
- Ticker animation continues during refresh; data updates in place.

### Files Changed
- `src/routes/market.js` — new market proxy endpoint (GET /api/market) with server-side caching
- `src/server.js` — register market route
- `public/admin/index.html` — Market Pulse card rewritten with ticker HTML, CSS animation, single fetch to /api/market
